PS.1 - Protect Software From Unauthorized Access and Tempering

The task E SBOM pipeline demonstartes PS.1 by moving build and audit execution from an uncontrolled developer workstation into a controlled developer workstation into a controlled CI enviroment. In generate-sbom.yml; the workflow is triggered by repository push events and runs on an ephemeral ubuntu-latest runner. This means the container build and SBOM generation are tied to version-controlled code and executed in a clean, short-lived runtime instead of a persistent local machine. The succesful workflow run evidence confirms the process is repeatable and tamper-evident at the build stage. In practical terms, this reduces the risk of undocumented local modification entering release artifacts and strengthens confidence that the genrated Sbom reflects the commited source state.

PW.4 - Maitain Provenance Data for Software Components

Task E satisfies PW.4 by producing a machine-readable CycloneDX SBOM artifact on every successful workflow run and preserving it as downloadable audit evidence. The SBOM contains component-level provenance data (library identity and version information) that enables direct traceability of what was included in the built container at that commit point. This is operationally significant because it converts supply-chain visibility from assumption into verifiable records. When new CVEs emerge, the team can rapidly determine exposure by matching affected package identifiers and versions against the stored SBOM artifact, rather than relying on manual recollection or static dependency files alone

RV.1 - Identify and Confirm Vulnerabilities on an Ongoing Basis

The workflow’s use of cyclonedx-json is a deliberate engineering choice that supports RV.1. A machine-readable SBOM can be consumed automatically by vulnerability-management tooling, enabling continuous and repeatable reassessment as vulnerability intelligence changes. Unlike static narrative documentation, JSON-formatted SBOM output is structured for integration into scanning pipelines and security platforms. Because the SBOM is regenerated on code changes and can be rescanned without rebuilding the full analysis narrative, the process supports ongoing identification and confirmation of newly disclosed component vulnerabilities
